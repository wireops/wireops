# wireops v1.0 – Expected Scope

## 1. Product Definition

wireops v1.0 is a declarative Git-driven control plane for Docker Compose.

It provides:

- Multi-host stack synchronization
- Versioned deployments
- Idempotent reconciliation
- Rollback support
- Drift detection
- Safe orphan detection

It does **not** aim to be a container orchestrator.

---

# 2. Core Architecture (v1)


Git
↓
Server (Control Plane)
↓
Agents
↓
Docker Engine


### Responsibilities

## Server (Control Plane)

- Poll Git repositories (cron-based)
- Trigger sync via webhook (optional)
- Render compose files
- Inject required wireops labels
- Generate override files
- Compute checksum of rendered compose
- Increment stack version
- Store stack revisions
- Deliver desired state to agents
- Track agent status

## Agent

- Poll server (fixed interval or long polling)
- Send applied versions per stack
- Receive tasks only if version changed
- Apply compose via Docker CLI
- Persist local applied state
- Report sync status
- Detect drift via labels
- Detect orphan stacks

---

# 3. Functional Scope (IN)

## 3.1 Git Integration

- Repository URL per stack
- Branch or ref selection
- Compose path selection
- Cron-based polling
- Webhook trigger (optional)

---

## 3.2 Render Engine (Server-Side)

- Clone repository
- Read compose file
- Merge override file
- Inject labels
- Generate final rendered compose
- Compute SHA256 checksum
- Persist revision history

Original compose file is never modified.

---

## 3.3 Versioning Model

Each stack must include:

- `version` (monotonic integer)
- `desired_commit`
- `checksum`

Version is incremented when:

- New commit detected
- Rollback performed
- Structural configuration changes

Rollback creates a new version.

---

## 3.4 Agent Reconciliation

Agent logic:

If:


desired_version > applied_version


Then:

- Apply rendered compose
- Update local state
- Report success/failure

Otherwise:

- Do nothing

Reconciliation must be idempotent.

---

## 3.5 Labeling Contract

All managed services must include:

- `wireops.managed=true`
- `wireops.stack_id`
- `wireops.version`
- `wireops.commit`
- `wireops.checksum`
- `wireops.agent_id`

Labels enable:

- Drift detection
- Orphan detection
- Safe garbage handling
- Debugging

---

## 3.6 Drift Detection

A stack is marked DRIFTED when:

- Container labels do not match desired version
- Commit differs from expected
- Checksum mismatch

Drift detection does not auto-heal in v1.
It surfaces state clearly.

---

## 3.7 Orphan Detection

An orphan stack is defined as:

- `wireops.managed=true`
- Stack not registered in control plane

Policy for v1:

- Detect and mark
- Manual removal only

No automatic deletion in v1.

---

## 3.8 Rollback

User can select a previous revision.

Server:

- Sets desired_commit to previous commit
- Increments version
- Triggers reconciliation

Rollback is treated as a new desired state.

---

## 3.9 Disaster Recovery

Supported model:

- New CA generation allowed
- Agents re-bootstrap
- System converges from Git
- No dependency on previous CA

DR must result in clean convergence.

---

## 3.10 Security Model

- mTLS between server and agents
- Certificate-based identity per agent
- No Git credentials on agents
- Server renders compose and delivers it

---

# 4. Non-Goals (OUT OF SCOPE FOR v1)

The following are explicitly excluded:

- Container scheduling
- Rolling update orchestration
- Horizontal auto-scaling
- Service mesh
- Secret management system
- Distributed networking layer
- UI-based compose editing
- Placement algorithms
- Multi-cluster federation
- Automatic health-based restart loops
- Full observability stack

If a feature resembles Kubernetes functionality, it is out of scope.

---

# 5. Operational Guarantees (v1)

wireops v1 guarantees:

- Deterministic reconciliation
- Version-based sync
- Idempotent restarts
- Multi-agent consistency
- Explicit ownership via labels
- Recoverability via Git
- Safe rollback
- Clear stack state visibility

---

# 6. Completion Criteria for v1

wireops v1.0 is considered complete when:

- [ ] Version increment logic works reliably
- [ ] Agent applies only when version changes
- [ ] Rollback works via revision history
- [ ] Drift detection is functional
- [ ] Orphan detection is functional
- [ ] Override file is used (no compose mutation)
- [ ] Multi-agent operation is stable
- [ ] Restart of server or agent preserves convergence
- [ ] Disaster recovery scenario tested
- [ ] Labels are consistently applied to all managed services

---

# 7. Definition of Done (Strategic)

wireops v1 must feel:

- Predictable
- Deterministic
- Declarative
- Safe
- Simple

If the system begins to require orchestration logic
instead of reconciliation logic,
the scope has exceeded v1.

---

# 8. Final Positioning

wireops v1 is:

> GitOps for Docker Compose, without Kubernetes complexity.

It stops at declarative multi-host synchronization.

And it does that extremely well.
