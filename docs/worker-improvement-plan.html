WireOps Worker · Improvement Plan

Plano de Evolução do Worker

Planejamento focado em confiabilidade de entrega, shutdown gracioso, semântica correta de erro, observabilidade operacional e simplificação estrutural do worker, incluindo versionamento via build args e entrega do binário único na pipeline de produção.

5 frentes principais de melhoria
4 fases sugeridas de implementação
1 binário único do worker, sem frontend
Objetivo

O que queremos resolver

  • Eliminar perda de resultados em reconnect, restart e shutdown.
  • Fazer o worker encerrar drenando trabalho sem aceitar novas execuções.
  • Diferenciar erro operacional de estado de negócio no protocolo.
  • Medir de forma real conexão, backlog, overload e tempo de execução.
  • Reduzir duplicação no transporte e formalizar a versão do worker no build.
Ordem Recomendada

Sequência de ataque

  • 1. Entrega confiável de resultados
  • 2. Shutdown gracioso real
  • 3. Semântica correta de erro
  • 4. Observabilidade operacional
  • 5. Simplificação estrutural e release engineering
Frente 1

Entrega confiável de resultados

Direção

Substituir as filas em memória por um spool local persistente em WORKER_STACK_DIR, garantindo replay após reconnect e sem apagar a mensagem até o server confirmar recebimento.

Estrutura sugerida

WORKER_STACK_DIR/ spool/ outbox/ pending/ sent/ meta/

Formato do evento

  • message_id
  • kind: command_result ou job_completed
  • command_id ou job_run_id
  • created_at
  • attempts
  • payload com envelope completo

Mudanças de protocolo

  • Adicionar message_id em CommandResult e JobCompletedMessage.
  • Adicionar uma mensagem de ACK do server para o worker, por exemplo MsgAck.
  • Fazer o replay por ordem temporal e deduplicação por message_id.

Regras operacionais

  • O worker grava no spool antes de tentar enviar.
  • O server processa de forma idempotente por message_id.
  • A mensagem só sai do spool após ACK explícito.
  • No reconnect, o worker reenvia tudo que estiver em pending.
  • O spool persiste apenas mensagens de saída, nunca comandos de entrada com .env ou RunJobCommand.Env.
Primeira versão simples: manter só pending e remover o arquivo após ACK. Isso já elimina a perda de resultado em boa parte dos cenários sem exigir um estado mais complexo.

Proteção do spool

  • Sem nova chave operacional dedicada.
  • O worker deriva a chave local do spool a partir do WORKER_TOKEN.
  • Usar derivação determinística local, por exemplo via HKDF com contexto do spool.
  • Diretório com permissão 0700 e arquivos com 0600.
  • Se o token mudar e um spool legado não puder ser lido, o worker deve mover para quarentena ou descartar com métrica e log administrativo.
Frente 2

Shutdown gracioso real

Problema atual

O worker fecha a conexão cedo demais, drena depois e corre o risco de sair com resultados ainda não confirmados pelo server.

Novo comportamento

  • Marcar o worker como draining.
  • Parar de aceitar novos comandos executáveis.
  • Manter o websocket ativo.
  • Continuar heartbeats com estado draining=true.
  • Esperar jobs e comandos ativos terminarem.
  • Flush do spool pendente.
  • Aguardar ACKs até um timeout configurável.
  • Fechar o websocket só no final.

Revogação do worker

  • Ao receber revogação, o worker para definitivamente de aceitar trabalho.
  • Remove o diretório WORKER_STACK_DIR/spool/ antes de encerrar.
  • Opcionalmente limpa também workdirs temporários remanescentes em stacks/*/cmd-*.
  • Se a revogação for percebida só no próximo boot ou reconnect, o cleanup roda antes do encerramento final.

Fallback

Se a conexão já tiver caído, o worker pode abrir uma sessão curta de reconnect apenas para publicar o que ficou no spool.

Política durante draining

  • Recusar novos deploys, jobs e operações interativas com erro claro: worker draining.
  • Permitir ACKs e mensagens de controle estritamente necessárias.
Frente 3

Erro operacional vs estado de negócio

Endpoints e comandos como probe, inspect, get_status e get_resources não devem mascarar falha de Docker ou Compose como se fosse ausência legítima de recurso.

Diretriz

  • Falha real de infraestrutura deve preencher CommandResult.Error.
  • Quando houver payload parcial útil, incluir degraded=true e source_error.
  • Não inferir “stack ausente” a partir de erro operacional.

Frente 5

Redução de acoplamento e duplicação

Transporte único

Consolidar a implementação websocket em worker/transport e descontinuar duplicações.

Registry de handlers

Trocar o switch crescente por um mapa de handlers com semáforo e policy de draining.

Interfaces menores

Introduzir contratos claros para Outbox, Connection, Dispatcher e SessionState.

Release Engineering

Versão via build args e entrega do binário único

Build info

Criar um pacote de build info com valores default para desenvolvimento e substituição via ldflags na pipeline.

package buildinfo var Version = "dev" var Commit = "none" var BuildDate = "unknown"

Exemplo de build

go build -ldflags="-X github.com/wireops/wireops/internal/buildinfo.Version=${GIT_TAG} \ -X github.com/wireops/wireops/internal/buildinfo.Commit=${GIT_SHA} \ -X github.com/wireops/wireops/internal/buildinfo.BuildDate=${BUILD_DATE}"

Diretrizes da pipeline

  • Build do worker independente do frontend.
  • Release do worker disparada por tags de produção.
  • Não publicar binário standalone do worker como artifact.
  • Publicar apenas a imagem Docker do worker, separada da imagem do server.
  • Garantir que a imagem do worker contenha apenas o binário e dependências mínimas de runtime.
O worker deve ser entregue apenas como imagem Docker isolada. Sem pb_public, sem assets de frontend e sem acoplamento ao bundle do server.
Fases

Backlog de implementação

Fase 1 · Segurança operacional imediata

  • Spool persistente local
  • message_id no protocolo
  • ACK idempotente
  • Shutdown com drain antes de fechar conexão
  • Métricas reais de conexão e spool

Fase 2 · Correção semântica

  • Revisar probe, inspect, get_status e get_resources
  • Remover falso “vazio” em erro operacional
  • Adaptar server e UI para degraded e source_error

Fase 3 · Simplificação

  • Consolidar transporte websocket
  • Introduzir registry de handlers
  • Limpar código duplicado ou legado

Fase 4 · Release do worker

  • Versionamento via ldflags
  • Imagem Docker do worker por tag
  • Sem artifact binário do worker
  • Imagem minimalista do worker
  • Pipeline do worker independente do frontend
Arquitetura Alvo

Estrutura sugerida do worker

app · lifecycle, bootstrap e shutdown
transport · sessão websocket, ACKs e heartbeats
outbox · spool persistente e replay
dispatch · registry de handlers, semáforos e policy de draining
executor · operações Docker, Compose e jobs
telemetry · métricas, build info e estado do worker
Aceite

Critérios de aceite principais

  • Reconnect não perde CommandResult nem JobCompletedMessage.
  • Restart do processo não apaga mensagens pendentes.
  • Replay não duplica atualização no server.
  • SIGTERM durante deploy ou job preserva publicação do resultado.
  • UI distingue “não encontrado” de “erro de infraestrutura”.
  • Métricas refletem o estado real do worker.
Resultado esperado

Ganho prático

  • Menos divergência entre worker e control plane.
  • Menor risco de estado fantasma após reconnect ou shutdown.
  • Operação mais observável e auditável.
  • Pipeline mais previsível para distribuição do worker em produção.
  • Base de código mais sustentável para crescer novas capacidades.
